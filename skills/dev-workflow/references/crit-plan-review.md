# Crit Plan-Review Gate (Step 4)

Deep reference for Step 4's **crit gate path** (`plan_review_gate: "crit"`, opt-in). Read and follow this top to bottom whenever `plan_review_gate` resolves to `crit` and Step 4 has reached sub-step 2 path (b) — do **not** pre-gate reading it on `crit` availability or browser reachability. **This reference owns both the availability determination and the reachability determination**: step 1 below checks the `crit --version` exit code, and step 2 checks `CLAUDE_CODE_REMOTE`, returning `fallback` on either failure. This file is the single canonical home for the crit gate procedure; Step 4 delegates here rather than restating it. See § Procedure step 7 (Fallback contract) for where a `fallback` return leads.

Like the visual gate, this path runs **outside Plan Mode** (`plan_mode_active = false` whenever `plan_review_gate` is not `plan-mode`; see `SKILL.md` § Configuration), so it freely performs non-read-only operations (launching the external `crit` CLI). The canonical plan document is `.claude/plans/<slug>.md`, written by Step 4 sub-step 2 path (b) before this procedure runs.

This contract is based on hands-on testing against **crit 0.17.1** (Homebrew, 2026-07-09), not crit's published documentation alone (including crit's own published Claude Code integration skill, `tomasz-tomczyk/crit`). If crit's observed behavior diverges from what's described here, re-verify against your installed version and update this reference.

## crit CLI contract

crit (<https://github.com/tomasz-tomczyk/crit>) is a **separately-installed** local binary, not bundled with this skill. This gate only invokes it — it never installs, configures, or depends on crit's own Claude Code plugin / `/crit` slash command / `agent_cmd` / live-thread features. Its contract, as observed:

- **Availability**: `crit --version` exits `0` when installed. `command -v crit` is avoided as an availability probe — it is not covered by this skill's `Bash(crit *)` allowed-tools grant and would trip a permission dialog.
- **Invocation**: `crit plan --name <slug> "<file>"` — reviews the Markdown plan at `<file>`, saving versioned snapshots under `~/.crit/plans/<slug>/` (`v001.md`, `v002.md`, … — entirely outside the repository) and opening the local default browser as a new tab. **Use the `crit plan` subcommand, not bare `crit <file>`** — Step 0 smoke testing found bare `crit <file>` invocations repeatedly failed to daemonize in this workflow's execution environment (the daemon process never appeared, and the client exited non-zero), while `crit plan --name <slug>` reliably launched and stayed alive across every trial. No `--timeout` (or equivalent) flag exists on `crit plan` — see § Procedure step 6 (Abandonment note) for the resulting abandonment behavior.
- **Blocking**: the invoking process blocks until the user clicks "Finish Review" in the browser (or the gate is interrupted — see the non-zero-exit row below).
- **Exit codes and stdout, on completion**: **both** approve and revise-with-comments complete with **exit code `0`** — unlike the visual gate, crit's exit code does not distinguish the decision. The decision is read from stdout: a literal `approved: true` or `approved: false` line, followed (on `false`) by the unresolved-comment count and a JSON array of comment objects. **Non-zero exit** (`crit stop --all`, SIGINT / SIGTERM, or a launch failure) means the review did not complete normally.
- **Comment JSON shape** (from `crit comments --json` and the completion stdout): `{"scope": "line"|"review", "path": "<file>", "id": "<id>", "start_line": <n>, "end_line": <n>, "body": "<comment text>", "anchor": "<nearby heading/text>", "author": "<name>", "created_at": "<iso8601>", "updated_at": "<iso8601>", "review_round": <n>}`. A review-level comment (not tied to a specific line) omits `path` and `anchor` and carries `scope: "review"`. **`quote` (the reviewer's selected text) may appear per crit's own documentation** — treat it as optional, use-if-present, never required.
- **Round-trip versioning**: crit itself versions each `crit plan --name <slug>` invocation against the same file (`v001` → `v002` → …) and persists prior comments as unresolved across rounds — no `.plan-review.prev.md`-equivalent snapshot is needed on this path (contrast the visual gate's § Prev snapshot).
- **Repository footprint**: none. All crit state (versioned plan snapshots, comments, session metadata) lives under `~/.crit/plans/<slug>/` and `~/.crit/sessions/`, outside the repository — confirmed via `git status` during Step 0 smoke testing. No entry in `SKILL.md` § Workflow artifacts is needed for crit.

## Procedure

1. **Check availability.** Run `crit --version`. Non-zero exit (including command-not-found) → return **`fallback`** to Step 4 immediately, without launching.
2. **Detect local-browser reachability.** Run `printenv CLAUDE_CODE_REMOTE` (same check as `references/visual-plan-review.md` step 2). A value of `true` means the local browser is unreachable → return **`fallback`** to Step 4 immediately. Any other value (including unset) → continue.
3. **Launch the gate (background Bash).** Run `crit plan --name "<slug>" ".claude/plans/<slug>.md"` as a **background Bash** command (`run_in_background: true`) — the same mechanism the visual gate uses, not the `Agent` tool — and emit a Progress Visibility status line in the same turn. Do not pass `--no-open` (that flag is for headless verification only, not the real gate). `<slug>` is the same value Step 4 sub-step 2 path (b) resolved.
4. **Read the decision.** When the background process exits, read its exit code and captured stdout together (§ crit CLI contract above — the decision line and comment JSON, when present, appear on stdout).
5. **Decision mapping.** Evaluate in order (first match wins):
   - **Non-zero exit**, or no parseable `approved:` line obtained → return **`fallback`** to Step 4 (crit was cancelled, interrupted, or failed to launch — see § Fallback contract, step 7 below, for where this leads).
   - **`approved: true`** → maps to Step 4's `accept`. Any accompanying comments are **advisory only** (the user chose approve regardless) — record them but do not block. Return **`approve`** to Step 4; proceed directly to implementation (no `ExitPlanMode`, this path runs outside Plan Mode).
   - **`approved: false`** → apply every comment in the returned JSON array directly to `.claude/plans/<slug>.md` (there is no separate served-file copy on this path — crit's own versioning is the round-trip record). For each comment, resolve its target block: for a line-scoped comment (`scope: "line"`), use `anchor` + `start_line`/`end_line` as the primary locator (using `quote` instead, when present, for a more precise match); for a **review-level comment** (`scope: "review"`, no `path`/`anchor`/line range — a comment on the review as a whole rather than a specific line), there is no block to resolve — treat it as a plan-wide instruction and apply it to whichever section it describes (inferred from `body`), or to the plan's Overview/Risks if it does not name a specific section. Then apply the requested change autonomously — same discipline as Step 3 finding application. This comment-application happens for **every** `approved: false` result, before and regardless of the branch below. Then branch on the nature of the (already-applied) changes:
     - **Zero comments** (empty array): re-launch the gate (loop back to step 3, same `<slug>`) — crit's own round-versioning shows the user their prior submit had no effect.
     - **Approach-level material change** (a new overall approach, a scope change — same vocabulary as Step 4's `rewrite-approach` bucket): return **`rewrite-approach`** to Step 4, which adds Step 3-(N_plan+1), re-reviews the plan, and re-enters Step 4 from sub-step 1.
     - **Localized edits** (wording, a bounded section edit, a single Decision tweak): re-launch the gate (loop back to step 3) — the edits already landed in `.claude/plans/<slug>.md` above. crit has no structured "switch to alternative" toggle (comments are free-text `body` strings), so a comment requesting a Decision swap is a localized edit like any other — when applying it, propagate the dependent updates (Approach / Risks / Scope / Test-plan sweep) exactly as the text-path swap-decisions bucket does (`SKILL.md` Step 4 sub-step 3). The loop is human-paced (each iteration blocks on a fresh browser submit), so no machine iteration cap is needed, matching the visual gate's loop.
6. **Abandonment note.** `crit plan` has no `--timeout` equivalent (confirmed via `crit plan --help`), so a review left open in the browser with no interaction simply keeps the background process waiting — there is no eventual auto-fallback the way the visual gate's 24h `--timeout` default provides. An abandoned crit gate requires the user to explicitly interrupt it (closing out via `crit stop --all`, or Ctrl-C on the foreground terminal if run there) to produce a non-zero exit and fall through to `fallback`. This is an accepted limitation of the opt-in `crit` path, not a defect to work around with an added watchdog process.
7. **Fallback contract.** Return **`fallback`** to Step 4 on any of: `crit` unavailable (step 1), local browser unreachable (step 2), non-zero `crit plan` exit, or the launch Bash call being blocked / yielding no result. Step 4 then continues with the **visual gate** (`references/visual-plan-review.md`) per its own routing table — never straight to chat. A `fallback` here never silently stalls the workflow: the visual gate's own `fallback` (if it also triggers) still has the no-Plan-Mode chat-approval path to fall through to.

The three return values to Step 4 are **`approve`**, **`rewrite-approach`**, and **`fallback`** (routing defined in § Procedure step 5 above) — identical in shape to the visual gate's contract (`references/visual-plan-review.md`), so Step 4 handles both uniformly.
