# Crit Commit-Review Gate (Step 10)

Deep reference for Step 10's **crit diff-review path** (`commit_review_gate: "crit"`, opt-in). Read and follow this top to bottom whenever `commit_review_gate` resolves to `crit` and `references/interactive-commits.md` § Per-commit loop sub-step **a. Present** reaches its Diff element for a given commit. This file is the single canonical home for the crit commit-review procedure; `references/interactive-commits.md` delegates here rather than restating it.

This is a sibling of `references/crit-plan-review.md` (the `plan_review_gate: "crit"` gate at Step 4) — the two share the same crit CLI, the same background-Bash launch discipline, and the same "human-paced, no machine iteration cap" reasoning — but they are **not** the same config key (see `SKILL.md` § Configuration's `commit_review_gate` bullet for why) and their decision-mapping targets differ (Step 4 maps onto `accept` / `rewrite-approach` / `fallback`; this gate maps onto the existing per-commit `accept` / `adjust` / `cancel` disposition already defined in `references/interactive-commits.md`).

## crit CLI contract (file/dir review mode)

Shared facts already documented in `references/crit-plan-review.md` § crit CLI contract — not restated in full here: crit is a separately-installed local binary this gate only invokes (never installs/configures it, and never depends on its own Claude Code plugin / `/crit` slash command / `agent_cmd` / live-thread features); the invoking process blocks until the user finishes the review in the browser (or the gate is interrupted).

This gate uses crit's **file/dir review mode** (`crit <file-1> <file-2> ...`) — not the `crit plan` subcommand `references/crit-plan-review.md` uses for Markdown plan review. The two modes are different code paths with different contracts; do not assume `crit-plan-review.md`'s findings transfer here without the verification noted below.

- **Availability**: same probe as `references/crit-plan-review.md` § crit CLI contract — `crit --version` exits `0` when installed (`command -v crit` is avoided; see that reference for why).
- **Invocation**: `crit <file-1> <file-2> ...` — pass exactly the files belonging to the commit under review, relative paths, one argument per file, each double-quoted (same verbatim-from-`-z`-output recovery and quoting discipline `c. Stage` / `d. Commit` apply to this identical file list). **Do not pass `--no-open` or `--quiet`** — real-launch verification during this feature's plan review found these flags correlate with unreliable daemon startup; the plain form does not (see "Empirically verified" below).
- **Scope**: crit scopes its review to **exactly the files passed as arguments** — other uncommitted changes in the working tree (files belonging to other proposed commits in the same Step 10 run) are excluded. Untracked new files are picked up natively when passed as an argument; no `git add -N` intent-to-add preprocessing is needed (unlike some other diff-review workflows in this project).
- **Decision output — read both stdout and stderr**: crit's own source (`internal/daemon/client.go`) writes the `approved: true`/`approved: false` decision line to **stderr**, and crit's own published Claude Code integration skill confirms this ("read stdout and follow its instructions; check stderr for `approved: true` or `approved: false`"). `references/crit-plan-review.md` documents the decision as stdout-only for the `crit plan` subcommand — that claim is unverified for this gate's file/dir mode and may itself be stale (out of scope to fix here; see the "Not verified in a completed live round" paragraph below). **Capture and scan both streams** for the `approved:` line rather than assuming either one exclusively; do not gate the parse on a single stream.
- **Comment JSON shape**: same shape `references/crit-plan-review.md` § crit CLI contract documents — unverified in a completed live round for this gate's file/dir mode (see below).
- **Non-zero exit** (interrupted, closed without finishing, or a launch failure) means the review did not complete normally.

**Empirically verified**: the plain `crit <file>` invocation form (no `--no-open`/`--quiet`) reliably daemonizes, and correctly scopes to the specified file(s) with native untracked-file pickup. `crit-plan-review.md` needed `crit plan --name <slug>` instead of bare `crit <file>` because bare invocation was unreliable for *plan* review — that finding is specific to selecting the correct interpretation mode for a Markdown plan file, not a general daemonization problem; the plain `crit <files>` form used here was reliable.

**Not verified in a completed live round**: the exact JSON shape of `approved: false` comments in file/dir mode, and which stream actually carries the decision line in practice — verify both the first time this gate is exercised for real.

## Procedure

1. **Check availability and reachability — once per run, cached.** The first time Step 10 needs this gate (the first commit whose Diff element is reached with `commit_review_gate: "crit"`), run **both, unconditionally (no short-circuit — each is a cheap, sub-second probe, so run (b) even if (a) already failed)**: (a) `crit --version` (non-zero exit / command-not-found → unavailable) and (b) `printenv CLAUDE_CODE_REMOTE` (value `true` → local browser unreachable). Cache the combined result (`crit_commit_review_available`) for the rest of the run — do not re-check per commit. This check is **independent** of any `crit --version` / `CLAUDE_CODE_REMOTE` check `plan_review_gate: "crit"` may have already performed this session at Step 4: the two gates are separately-toggleable features consumed at very different points in the workflow (Step 4 vs. late Step 10, with Steps 5–9 intervening), and Step 4's own `visual` / `crit` gates already each independently re-check `CLAUDE_CODE_REMOTE` rather than sharing a cross-gate cache (`references/visual-plan-review.md`) — this gate follows that same established self-contained-check convention rather than threading cached state across the intervening Steps to save a single cheap, sub-second probe.
2. **If either check failed**: treat the entire Step 10 run as `commit_review_gate: "diff"` from this point on — every commit's Diff element renders via the existing chat-text presentation in `references/interactive-commits.md` § Per-commit loop sub-step **a. Present** (verbatim / condensed / skeleton). Emit **one** informational note for the whole run (not per commit), in the resolved `language` (see `SKILL.md` § Configuration's `language` bullet), before continuing this commit's `Present` rendering under that fallback: when unavailable, name the install URL (<https://github.com/tomasz-tomczyk/crit>); when remote (and not unavailable), name that no local browser is reachable; when both checks failed, name only the install URL — it is the more fundamental blocker, and fixing it is a prerequisite to the reachability check ever mattering. Do not re-attempt crit later in the same run.
3. **If both checks passed**: for the commit currently at sub-step **a. Present**'s Diff element, launch `crit <file-1> <file-2> ...` (the commit's file list, the same paths `c. Stage` will later pass to `git add --`) as **background Bash** (`run_in_background: true`) — not the `Agent` tool (mirrors `references/crit-plan-review.md`'s "background Bash, not `Agent`" discipline; this launch does not count against the three fixed dispatch sites named in `SKILL.md` § Configuration's `Agent` tool usage bullet). Emit a Progress Visibility status line in the same turn.
4. **Read the decision.** When the background process exits, read both captured stdout and stderr together and locate the `approved:` line in whichever stream carries it (see the stdout/stderr note above).
5. **Decision mapping** (first match wins):
   - **Non-zero exit, or no parseable `approved:` line obtained** → this commit only falls back to the existing chat-text Diff presentation and the standard § Approval token closed list accept/adjust/cancel gate (`references/interactive-commits.md` § Per-commit loop sub-step **b. Per-commit accept gate**). `crit_commit_review_available` is **not** cleared — crit is used again for the next commit.
   - **`approved: true`** → maps directly to this gate's `accept` — proceed straight to `c. Stage` / `d. Commit` for this commit. Any accompanying comments are advisory only (the user chose approve regardless).
   - **`approved: false`, comments empty** → re-launch crit on the same (unchanged) file set (per step 3's launch instructions above) — communicates that the submit had no effect.
   - **`approved: false`, comments present** → process every comment in the round in this fixed order: apply all `scope: "line"` edits, then re-run their required verification (below), then process `scope: "review"` dispositions last — they must run only after that re-verification completes, since some redirect control flow:
     - **`scope: "line"`** (a specific code-level fix request): apply the requested code edit. A round may contain more than one `scope: "line"` comment — apply all of them here; do not re-verify between individual comments.

       If this round applied at least one `scope: "line"` edit, re-run **Step 7 (Check / Test) and Step 7.5 (Rules Compliance Review) — no exceptions** once for the whole round (the same fix-and-reverify pattern Step 8 Code Review uses for its own reviewer-finding fixes — apply every finding from one iteration, then re-verify once: "Code fixes routinely introduce fresh bugs... the next review round is how those leaks get caught" applies equally here). No user gate for this re-verification — the AI applied its own fix(es) from specific review comments and re-verifies them itself, the same way Step 8 does. If Step 7 fails, follow Step 7's existing fail-stop discipline (retry, then report and stop). This is **not** `references/interactive-commits.md` § Mid-loop adjust branch **g** ("Actual-code changes made during the gate") — branch g exists for the user's own out-of-band manual edits discovered incidentally, a different scenario from an AI-applied fix driven by a specific parsed review comment. **This re-verification must complete before processing any `scope: "review"` comment below** — a `scope: "review"` disposition that redirects control flow (`cancel`, or an `adjust` branch re-entering a different gate) must never bypass the pending re-verification for this round's line edits.
     - **`scope: "review"`** (not tied to a line — e.g. a request to change the subject, split the commit, cancel, or any other non-code-content request; processed after the `scope: "line"` re-verification above): classify the comment's `body` text per `SKILL.md` § Approval token closed list's four buckets exactly as if the user had typed it in chat. A **`cancel`** classification routes straight to `references/interactive-commits.md` § Mid-loop cancel — the `Mid-loop adjust — closed-list branches` classifier (branches a–g) covers only the `adjust` bucket (its own entry condition is "any user response categorized as `adjust`"), so a cancel-shaped comment must not be forced through it. An **`adjust`** classification enters that classifier as normal and applies the matched branch's disposition. An **`accept`** classification is advisory only — like `approved: true`'s accompanying comments above, the round's own `approved: false` verdict already governs whether this commit lands, so a comment reading as an affirmation does not itself trigger `c. Stage` / `d. Commit`.
     After all comments are applied and any re-verification lands, decide whether to relaunch crit based on how this round's `scope: "review"` comment (if any) resolved: a **direct `cancel` classification** already stopped Step 10 outright via § Mid-loop cancel above — do not relaunch. An **`adjust` classification** hands off to its matched Mid-loop adjust branch's own next action instead (which may itself re-enter this commit's `Present` step, or resolve the commit via a merge/drop — `references/interactive-commits.md` § Mid-loop adjust branches **c**/**d** are an explicit normal completion, not a cancel) — do not also apply this sentence. Otherwise (no `scope: "review"` comment this round, or an `accept`-classified one, advisory only per above) the commit is still pending — re-launch crit (per step 3's launch instructions above) on the updated file set for another round.
6. **No machine iteration cap.** Like `references/crit-plan-review.md`, each round blocks on a real browser submit, so the loop is human-paced — no iteration ceiling is needed.

## Fallback contract

This gate never silently stalls Step 10: unavailability or an unreachable browser routes the whole run to `diff` mode (step 2's "If either check failed" case above); a single commit's crit interruption routes just that commit to the `diff` gate (step 5's non-zero-exit case) while crit stays in use for the rest of the run. Both destinations are the pre-existing chat-text Diff presentation — there is no intermediate browser-based fallback tier for this gate (unlike `plan_review_gate: "crit"`'s `crit` → `visual` → chat chain), since the chat presentation is already the gate's designed floor — it is the only surface guaranteed to work on Claude Code on the Web and other remote sandboxes where no browser gate can launch, so a new intermediate viewer would either be redundant with `crit` (when `crit` works) or unreachable itself (when it doesn't).
