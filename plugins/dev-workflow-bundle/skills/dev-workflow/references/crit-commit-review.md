# Crit Commit-Review Gate (Step 10)

Deep reference for Step 10's **crit diff-review path** (`commit_review_gate: "crit"`, opt-in). Read and follow this top to bottom whenever `commit_review_gate` resolves to `crit` and `references/interactive-commits.md` § Per-commit loop sub-step **a. Present** reaches its Diff element for a given commit. This file is the single canonical home for the crit commit-review procedure; `references/interactive-commits.md` delegates here rather than restating it.

This is a sibling of `references/crit-plan-review.md` (the `plan_review_gate: "crit"` gate at Step 4) — the two share the same crit CLI, the same background-Bash launch discipline, and the same "human-paced, no machine iteration cap" reasoning — but they are **not** the same config key (see `SKILL.md` § Configuration's `commit_review_gate` bullet for why) and their decision-mapping targets differ (Step 4 maps onto `accept` / `rewrite-approach` / `fallback`; this gate maps onto the existing per-commit `accept` / `adjust` / `cancel` disposition already defined in `references/interactive-commits.md`).

## crit CLI contract (commit range review mode)

Shared facts already documented in `references/crit-plan-review.md` § crit CLI contract — not restated in full here: crit is a separately-installed local binary this gate only invokes (never installs/configures it, and never depends on its own Claude Code plugin / `/crit` slash command / `agent_cmd` / live-thread features); the invoking process blocks until the user finishes the review in the browser (or the gate is interrupted).

This gate uses crit's **commit range review mode** (`crit --range <base>..<head>`) — not the `crit plan` subcommand `references/crit-plan-review.md` uses for Markdown plan review, and **not** the file/dir mode (`crit <file-1> <file-2> ...`) an earlier version of this gate used. **The file/dir mode was the root cause of a real bug**: it reviews the current on-disk content of the named files — crit's own runtime output says so explicitly ("file paths are intended for reviewing a small set of documents or plans. To review code changes, run `crit` with no arguments"). Passing file arguments never enters crit's git-diff-aware code path at all, so the browser showed only the final code, never a diff. Commit range mode is git-diff-aware and correctly renders the diff between the two given revisions.

- **Availability**: same probe as `references/crit-plan-review.md` § crit CLI contract — `crit --version` exits `0` when installed (`command -v crit` is avoided; see that reference for why).
- **Scoping mechanism — detached review object, no ref ever moves**: crit's commit-range mode diffs exactly the two revisions named in `--range`; it does not accept a file-list argument to additionally narrow which paths within that range are shown, and it ignores the working tree entirely (dirty or clean) outside the named range. To scope the review to **only** the commit currently at `a. Present`'s Diff element — while other not-yet-landed commits' files sit uncommitted in the working tree — this gate synthesizes a **plumbing-level commit object** for crit to diff against, without ever touching `HEAD`, a branch ref, or the working tree: stage just this commit's files into the index (`git add --`), snapshot the index into a tree object (`git write-tree` — reads the index only, so unstaged/untracked files never enter it; the resulting tree still carries every already-committed path unchanged from `HEAD` alongside the newly-staged ones — the review's file-level scoping comes from crit diffing `<pre_round_head>..<round_commit>` as two trees, which surfaces only the paths that actually differ, not from the snapshot itself omitting anything), then wrap that tree in a standalone commit object (`git commit-tree`, which writes to the object database but updates no ref and runs no hooks). crit reviews `<pre_round_head>..<round_commit>`; once the process ends, the index is unstaged back to its pre-round state (`git reset -- <files>`, itself index-only). Nothing is ever committed to a branch, so there is nothing to reset at the ref level and nothing that can be left stranded at `HEAD` if a session is interrupted mid-round — the round commit is a dangling object, invisible to `git log`, that git's normal garbage collection reclaims in the ordinary course of things.
- **Invocation**: `crit --range <pre_round_head>..<round_commit>` — no positional file arguments. `<pre_round_head>` is the `HEAD` SHA (`git rev-parse HEAD`) resolved once, the first round of a given commit's review (see step 3's per-round launch below for why it is not re-resolved every round); `<round_commit>` is this round's `git commit-tree` output. **Do not pass `--no-open` or `--quiet`** — same real-launch finding as `references/crit-plan-review.md`: these flags correlate with unreliable daemon startup; the plain form does not (see "Empirically verified" below).
- **Scope — verified to ignore the dirty working tree**: crit's `--range` mode reviews only the named commit range; other uncommitted changes sitting in the working tree (tracked or untracked, belonging to other not-yet-reviewed commits in the same Step 10 run) do not appear. Untracked new files belonging to *this* commit are picked up natively once staged; no `git add -N` intent-to-add preprocessing is needed.
- **Decision output — read both stdout and stderr**: crit's own source (`internal/daemon/client.go`) writes the `approved: true`/`approved: false` decision line to **stderr**, and crit's own published Claude Code integration skill confirms this ("read stdout and follow its instructions; check stderr for `approved: true` or `approved: false`"). `references/crit-plan-review.md` documents the decision as stdout-only for the `crit plan` subcommand — that claim is unverified for this gate's commit-range mode and may itself be stale (out of scope to fix here; see the "Not verified in a completed live round" paragraph below). **Capture and scan both streams** for the `approved:` line rather than assuming either one exclusively; do not gate the parse on a single stream.
- **Comment JSON shape**: same shape `references/crit-plan-review.md` § crit CLI contract documents — unverified in a completed live round for this gate's commit-range mode (see below).
- **Non-zero exit** (interrupted, closed without finishing, or a launch failure) means the review did not complete normally.

**Empirically verified**: `crit --range <base>..<head>` (crit v0.17.1) ignores the dirty working tree and reviews only the named commit range, and — the load-bearing fact this design depends on — **accepts a dangling (non-ref-reachable) `git commit-tree` object as `--range`'s `<head>`**, reporting exactly the staged paths and nothing else. `--range` is documented (`crit --help` lists it as "Review a commit range"), unlike the file/dir mode's undocumented content-only behavior. The stage → `write-tree` → `commit-tree` → `reset --` round-trip is clean: `write-tree` reflects only the staged paths, and the final `reset --` returns the index to `HEAD`'s own tree exactly, indistinguishable from never having run.

**Not verified in a completed live round**: the exact JSON shape of `approved: false` comments in commit-range mode, and which stream actually carries the decision line in practice — verify both the first time this gate is exercised for real.

## Procedure

1. **Check availability and reachability — once per run, cached.** The first time Step 10 needs this gate (the first commit whose Diff element is reached with `commit_review_gate: "crit"`), run **both, unconditionally (no short-circuit — each is a cheap, sub-second probe, so run (b) even if (a) already failed)**: (a) `crit --version` (non-zero exit / command-not-found → unavailable) and (b) `printenv CLAUDE_CODE_REMOTE` (value `true` → local browser unreachable). Cache the combined result (`crit_commit_review_available`) for the rest of the run — do not re-check per commit. This check is **independent** of any `crit --version` / `CLAUDE_CODE_REMOTE` check `plan_review_gate: "crit"` may have already performed this session at Step 4: the two gates are separately-toggleable features consumed at very different points in the workflow (Step 4 vs. late Step 10, with Steps 5–9 intervening), and Step 4's own `visual` / `crit` gates already each independently re-check `CLAUDE_CODE_REMOTE` rather than sharing a cross-gate cache (`references/visual-plan-review.md`) — this gate follows that same established self-contained-check convention rather than threading cached state across the intervening Steps to save a single cheap, sub-second probe.

2. **If either check failed**: treat the entire Step 10 run as `commit_review_gate: "diff"` from this point on — every commit's Diff element renders via the existing chat-text presentation in `references/interactive-commits.md` § Per-commit loop sub-step **a. Present** (verbatim / condensed / skeleton). Emit **one** informational note for the whole run (not per commit), in the resolved `language` (see `SKILL.md` § Configuration's `language` bullet), before continuing this commit's `Present` rendering under that fallback: when unavailable, name the install URL (<https://github.com/tomasz-tomczyk/crit>); when remote (and not unavailable), name that no local browser is reachable; when both checks failed, name only the install URL — it is the more fundamental blocker, and fixing it is a prerequisite to the reachability check ever mattering. Do not re-attempt crit later in the same run.

3. **If both checks passed — per-round launch**: for the commit currently at sub-step **a. Present**'s Diff element, resolve `<pre_round_head>` via `git rev-parse HEAD` **once — the first round of this commit's review** — and reuse that cached value for every later round of the *same* commit (no other commit can land while this commit's own review is still open, so `HEAD` cannot have moved in between; re-resolve only when this commit's review concludes and the per-commit loop advances to the next commit, whose own first round resolves a fresh value). For each round:
   1. Stage this commit's files (`git add -- <file-1> <file-2> ...`, the same paths `c. Stage` will later pass to the real commit).
   2. `git write-tree` to snapshot the now-staged index into a tree object, then `git commit-tree <tree> -p <pre_round_head>` (message piped via stdin — reuse the subject/body already fixed for this commit at sub-step **a. Present**, already resident in orchestrator state at this point in the loop, not re-derived from `references/interactive-commits.md`; this is the same content `d. Commit` would use) to produce `<round_commit>` — a dangling object; no ref moves, no hook runs. If staging or either git call fails, this round — and this commit — falls back to the `diff` gate (same established per-commit fallback pattern as an unreachable/interrupted crit launch); `crit_commit_review_available` is untouched, so crit stays in use for the remaining commits in the run. No explicit unstage is needed here even when `git add --` already succeeded before `write-tree` or `commit-tree` failed: leaving those files staged is harmless for the same reason step 4's unstage-failure case is (`c. Stage`'s later `git add --` and `d. Commit`'s pathspec-scoped commit behave identically whether or not these paths were already staged).
   3. Launch `crit --range <pre_round_head>..<round_commit>` as **background Bash** (`run_in_background: true`) — not the `Agent` tool (mirrors `references/crit-plan-review.md`'s "background Bash, not `Agent`" discipline; this launch does not count against the three fixed dispatch sites named in `SKILL.md` § Configuration's `Agent` tool usage bullet). Emit a Progress Visibility status line in the same turn.

4. **Read the decision, then unstage.** When the background process exits, read both captured stdout and stderr together and locate the `approved:` line in whichever stream carries it (see the stdout/stderr note above) — a non-zero exit or no parseable line is itself one of the outcomes step 5's decision mapping enumerates below, not an error at this step. **Regardless of what was read**, run `git reset -- <file-1> <file-2> ...` (the same pathspec staged in step 3's per-round launch sub-step 1's staging action) to return the index to its pre-round state. If this unstage fails, retry once, then treat it as **non-fatal** — the same per-item Progress you'd give any recoverable hiccup, per this project's canonical retry-once external-command-failure pattern (`references/interactive-commits.md` § Per-commit loop sub-step **e. Non-zero-exit retry (commit-attempt failure)**, adapted here: retry once after a 1–2 second sleep; if it still fails, render a one-line note with `<reason>` — the last non-empty line of stderr, truncated to ≤ 80 characters, or `(no stderr)` when empty — and proceed anyway. **Do not auto-recover** (no `git reset --hard`, no manual index/ref manipulation) — a failed unstage just leaves the files staged, which is harmless). Unlike `d. Commit`'s own retry failure, both `c. Stage`'s later `git add --` and `d. Commit`'s pathspec-scoped commit behave identically whether or not these paths were already staged.

5. **Decision mapping** (first match wins):
   - **Non-zero exit, or no parseable `approved:` line obtained** → this commit only falls back to the existing chat-text Diff presentation and the standard § Approval token closed list accept/adjust/cancel gate (`references/interactive-commits.md` § Per-commit loop sub-step **b. Per-commit accept gate**). `crit_commit_review_available` is **not** cleared — crit is used again for the next commit.
   - **`approved: true`** → maps directly to this gate's `accept` — proceed straight to `c. Stage` / `d. Commit` for this commit. Any accompanying comments are advisory only (the user chose approve regardless).
   - **`approved: false`, comments empty** → start a new round at step 3's per-round launch (rebuild `<round_commit>` from the same, unchanged file set) — communicates that the submit had no effect.
   - **`approved: false`, comments present** → process every comment in the round in this fixed order: apply all `scope: "line"` edits, then re-run their required verification (below), then process `scope: "review"` dispositions last — they must run only after that re-verification completes, since some redirect control flow:
     - **`scope: "line"`** (a specific code-level fix request): apply the requested code edit. A round may contain more than one `scope: "line"` comment — apply all of them here; do not re-verify between individual comments.

       If this round applied at least one `scope: "line"` edit, re-run **Step 7 (Check / Test) and Step 7.5 (Rules Compliance Review) — no exceptions** once for the whole round (the same fix-and-reverify pattern Step 8 Code Review uses for its own reviewer-finding fixes — apply every finding from one iteration, then re-verify once: "Code fixes routinely introduce fresh bugs... the next review round is how those leaks get caught" applies equally here). No user gate for this re-verification — the AI applied its own fix(es) from specific review comments and re-verifies them itself, the same way Step 8 does. If Step 7 fails, follow Step 7's existing fail-stop discipline (retry, then report and stop). This is **not** `references/interactive-commits.md` § Mid-loop adjust branch **g** ("Actual-code changes made during the gate") — branch g exists for the user's own out-of-band manual edits discovered incidentally, a different scenario from an AI-applied fix driven by a specific parsed review comment. **This re-verification must complete before processing any `scope: "review"` comment below** — a `scope: "review"` disposition that redirects control flow (`cancel`, or an `adjust` branch re-entering a different gate) must never bypass the pending re-verification for this round's line edits.
     - **`scope: "review"`** (not tied to a line — e.g. a request to change the subject, split the commit, cancel, or any other non-code-content request; processed after the `scope: "line"` re-verification above): classify the comment's `body` text per `SKILL.md` § Approval token closed list's four buckets exactly as if the user had typed it in chat (that list's fourth bucket — non-committal/interrogative wording — collapses into `adjust` by its own definition, so this classifier only ever needs to branch on the three resulting outcomes below: `cancel` / `adjust` / `accept`). A **`cancel`** classification routes straight to `references/interactive-commits.md` § Mid-loop cancel — the `Mid-loop adjust — closed-list branches` classifier (branches a–g) covers only the `adjust` bucket (its own entry condition is "any user response categorized as `adjust`"), so a cancel-shaped comment must not be forced through it. An **`adjust`** classification enters that classifier as normal and applies the matched branch's disposition — a branch that touches *other* commits' files (merge / split / reorder) operates on the working tree exactly as `git status` reports it, since no other commit's files were ever staged or otherwise touched by this gate. An **`accept`** classification is advisory only — like `approved: true`'s accompanying comments above, the round's own `approved: false` verdict already governs whether this commit lands, so a comment reading as an affirmation does not itself trigger `c. Stage` / `d. Commit`. **When a round contains more than one `scope: "review"` comment**, resolve by priority rather than arrival order — `cancel` > `adjust` > `accept`: a `cancel` classification anywhere in the round wins outright regardless of what else was submitted; absent a `cancel`, the first `adjust`-classified comment wins and any further `scope: "review"` comments in the round are not separately processed; if only `accept`-classified comments are present, all are advisory per above. This is a **priority-tier resolution** (rank order among the three buckets, not arrival order among the comments) — a distinct axis from the list-position-based first-match-wins discipline the rest of this gate's decision mapping uses; do not conflate "first match in this priority ranking" with "first comment to arrive in the round."
     After all comments are applied and any re-verification lands, decide whether to start a new round based on how this round's `scope: "review"` comment (if any) resolved: a **direct `cancel` classification** already stopped Step 10 outright via § Mid-loop cancel above — do not start a new round. An **`adjust` classification** hands off to its matched Mid-loop adjust branch's own next action instead (which may itself re-enter this commit's `Present` step, or resolve the commit via a merge/drop — `references/interactive-commits.md` § Mid-loop adjust branches **c**/**d** are an explicit normal completion, not a cancel) — do not also apply this sentence. Otherwise (no `scope: "review"` comment this round, or an `accept`-classified one, advisory only per above) the commit is still pending — start a new round at step 3's per-round launch (reusing the same `<pre_round_head>`, rebuilding `<round_commit>` from the now-edited files).
6. **No machine iteration cap.** Like `references/crit-plan-review.md`, each round blocks on a real browser submit, so the loop is human-paced — no iteration ceiling is needed.

## Fallback contract

This gate never silently stalls Step 10: unavailability or an unreachable browser routes the whole run to `diff` mode (step 2's "If either check failed" case above); a single commit's crit interruption routes just that commit to the `diff` gate (step 5's non-zero-exit case), and a staging/`write-tree`/`commit-tree` failure does the same (step 3's per-round launch, sub-step 2's git-call-failure case) — while crit stays in use for the rest of the run in both cases. Both `diff`-mode destinations are the pre-existing chat-text Diff presentation — there is no intermediate browser-based fallback tier for this gate (unlike `plan_review_gate: "crit"`'s `crit` → `visual` → chat chain), since the chat presentation is already the gate's designed floor — it is the only surface guaranteed to work on Claude Code on the Web and other remote sandboxes where no browser gate can launch, so a new intermediate viewer would either be redundant with `crit` (when `crit` works) or unreachable itself (when it doesn't). A step 4 unstage failure is the one non-fatal exception to this "route away" pattern — it does not fall back or stop, since nothing was ever committed to a ref for it to leave behind (see step 4's unstage handling).

**Browser-side scope-toggle caveat (not empirically verified — see the "Empirically verified" / "Not verified in a completed live round" paragraphs above for this file's verification-labeling convention)**: crit's browser UI is understood to expose a scope toggle (`all` / `unstaged`, and similar values depending on repo state) that a user could manually switch during a review round. If so, the default view — the commit range named by `--range` — is correctly scoped to just this commit; only a manual toggle away from that default could surface other commits' still-uncommitted working-tree content. This would be a residual, opt-in-only exposure, not a default-view leak.
